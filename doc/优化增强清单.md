### 测试问题总结

#### 问题1：双击后没有光标
- **现象**：双击文本元素后，`contenteditable` 被设置，但光标没有出现，无法输入文字
- **原因**：原实现中有 `mousedown` 事件处理器调用了 `e.preventDefault()`，阻止了默认的文本选择行为；另外 `range.selectNodeContents()` 配合 `window.getSelection()` 的选中文本操作在某些情况下与 `contenteditable` 冲突
- **解决**：简化 `attachTextEditHandlers` 中的 dblclick 处理器，只设置 `contenteditable="true"` 和调用 `focus()`，移除 `mousedown` 拦截和文本选择逻辑

#### 问题2：编辑后点击外部/blur时消息未保存
- **现象**：控制台显示 `[edit-bridge] blur, sending text-change:` 日志正常输出，但文件没有保存成功
- **原因**：FileViewer 中的 `onMessage` 函数有 `ev.source !== iframeRef.current?.contentWindow` 检查，由于 iframe 使用 `sandbox="allow-scripts"` 运行，与父页面非同源，导致 `ev.source` 比较失败，消息被直接丢弃
- **解决**：修改 FileViewer.tsx 中的消息来源检查逻辑，对 `od-edit-*` 类型消息（`od-edit-targets`、`od-edit-select`、`od-edit-reorder`、`od-edit-text-change`）放宽来源限制，允许来自 sandboxed iframe 的消息通过

#### 问题3：Docker构建失败
- **现象**：Docker构建在 `corepack enable` 步骤失败
- **原因**：网络问题导致 corepack 准备 pnpm 时超时
- **解决**：重试构建通常可以成功

## 2026-05-25: 页面刷新后 404 问题

### 概述
在 Docker 容器中运行 Open Design 时，首次打开项目文件页面可以正常访问，但按 F5 刷新后返回 404 错误。

### 问题描述
- **URL**: `http://127.0.0.1:7456/projects/3b6c6272-2afa-449c-b005-0c0376894410/conversations/a81fde1c-3dea-4cc5-a95f-b8fe89e4810c/files/mpc-platform-prototype.html`
- **首次访问**: 可以正常加载页面
- **F5 刷新**: 返回 `404 (Not Found)`
- **错误日志**: `GET ... 404 (Not Found)`

### 影响范围
- 项目文件页面的直接访问（通过 `/projects/...` 路由）
- 可能影响所有项目文件的直接链接访问

### 可能原因
1. **Next.js 静态文件服务**: `server.ts` 中使用 `express.static(STATIC_DIR)` 提供静态文件，但 `STATIC_DIR` 是 Next.js 的客户端构建产物，不包含服务器端路由处理
2. **路由结构不匹配**:
   - `/projects/...` 是 Next.js 客户端路由，需要 Next.js 服务器处理
   - `/api/projects/.../raw/...` 是后端 API 路由
   - 刷新时浏览器直接请求 `/projects/...` 路径，但 Express 找不到对应路由
3. **SPA fallback 缺失**: Express 静态文件服务没有配置 SPA fallback，将所有未匹配请求重定向到 `index.html`

### 相关文件
- `apps/daemon/src/server.ts` - Express 服务器配置
- `deploy/Dockerfile` - Docker 构建配置

### 修复方案
1. 使用中间件替代 `app.get('*')` 作为 SPA fallback，对 `/api` 路径跳过处理
2. 中间件在非 `/api` 路径且没有其他路由匹配时返回 `index.html`

### 修复代码（server.ts）
```typescript
// SPA fallback: serve index.html for any GET request that didn't match a static file or API route.
// This enables direct access to client-side routes like /projects/... on refresh.
// NOTE: This MUST be registered AFTER all API routes including connector routes.
if (fs.existsSync(STATIC_DIR)) {
  app.use((req, res, next) => {
    // Only apply SPA fallback to non-API routes
    if (req.path.startsWith('/api')) {
      return next();
    }
    const indexPath = path.join(STATIC_DIR, 'index.html');
    if (fs.existsSync(indexPath)) {
      return res.sendFile(indexPath);
    }
    next();
  });
}
```

### 验证结果
- `curl http://127.0.0.1:7456/api/health` 返回 `{"ok":true,"version":"0.7.0"}`
- `curl http://127.0.0.1:7456/api/connectors/composio/config` 返回 `{"configured":false,"apiKeyTail":""}`
- 直接访问 `http://127.0.0.1:7456/projects/...` 路由刷新后能正常加载（不再返回 404）

### 验证状态
- [x] API 路由不被 SPA fallback 拦截
- [x] F5 刷新后项目文件页面正常加载（不再返回 404）

## 2026-05-25: 拖动排序元素跑到父容器外面的Bug修复

### 概述
修复在编辑模式下拖动元素时，元素被移到错误位置（父容器外面）的问题。

### 问题描述
- **现象**：用户拖动"平台可用性"（在 hero-stats 内）到"覆盖行业领域"（也在 hero-stats 内）之后，元素跑到 hero-stats 外面作为 hero-content 的子元素
- **原因**：`moveElement` 函数中，当 `afterId` 指向的元素是拖动元素的祖先时，`newParent` 被设为祖先的父元素而不是祖先本身，导致元素被移到错误位置

### 修改文件
1. `apps/web/src/edit-mode/source-patches.ts`
   - 在 `afterId !== null` 分支中添加 `anchor.contains(el)` 检查
   - 如果 anchor 包含 el（el 是 anchor 的后代），则将 el 追加到 anchor 内部作为最后一个子元素
   - 否则按原有逻辑，将 el 作为 anchor 的 sibling 插入

### 功能说明
- **祖先检测**：使用 `anchor.contains(el)` 检测被拖动元素是否在目标元素内部
- **正确插入**：如果被拖动元素在目标元素内部，保持在同一父容器内移动；否则作为 sibling 插入
- **保持 DOM 层级**：确保拖动排序不会意外改变元素的 DOM 层级

### 验证标准
1. 在编辑模式下拖动 hero-stats 内的元素到同容器的另一个元素之后，元素应保持在 hero-stats 内部
2. 拖动后的元素路径（`data-od-source-path`）应正确反映新位置
