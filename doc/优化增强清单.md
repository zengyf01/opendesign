## 2026-05-20: HTML文件编辑及拖动功能

### 概述
实现了HTML文件的编辑模式功能，包括元素拖动排序和文本编辑。

### 修改文件
1. `apps/web/src/edit-mode/types.ts`
   - 新增 `ManualEditReorderMessage` 接口（`od-edit-reorder` 消息类型）
   - 新增 `ManualEditTextChangeMessage` 接口（`od-edit-text-change` 消息类型）
   - 新增 `move-element` patch 类型到 `ManualEditPatch` 联合类型

2. `apps/web/src/edit-mode/source-patches.ts`
   - 新增 `moveElement()` 函数：处理HTML元素的移动操作
   - 在 `applyManualEditPatch()` 中添加对 `move-element` patch 的处理

3. `apps/web/src/edit-mode/bridge.ts`
   - 新增拖拽相关函数：`createDropIndicator()`, `showDropIndicator()`, `hideDropIndicator()`, `clearDragSource()`, `startDrag()`, `endDrag()`, `attachDragHandlers()`, `setupDragAndDrop()`
   - 新增 `draggedId` 和 `dropIndicator` 变量用于追踪拖拽状态
   - 实现 `dragstart`, `dragend`, `dragover`, `dragleave`, `drop` 事件处理
   - 拖拽时显示蓝色指示线，根据鼠标位置判断插入点
   - 新增 `[data-od-drag-source]` 样式：拖拽源元素半透明显示

4. `apps/web/src/components/FileViewer.tsx`
   - 在 `onMessage` 处理函数中添加对 `od-edit-reorder` 和 `od-edit-text-change` 消息的处理

### 功能说明
- **拖动排序**：在编辑模式下，可拖动元素到新位置，蓝色指示线显示插入位置
- **文本编辑**：双击元素进入文本编辑模式，编辑后自动保存
- **循环检测**：防止将元素拖入其后代元素，避免DOM循环错误
- **视觉反馈**：拖拽源元素显示半透明效果，目标位置显示蓝色指示线


### 验证标准
1. **拖动排序**：进入编辑模式后，可拖动带 `data-od-id` 的元素，蓝色指示线显示插入位置，释放后元素顺序更新并保存

### 测试问题总结

#### 问题1：双击后没有光标
- **现象**：双击文本元素后，`contenteditable` 被设置，但光标没有出现，无法输入文字
- **原因**：原实现中有 `mousedown` 事件处理器调用了 `e.preventDefault()`，阻止了默认的文本选择行为；另外 `range.selectNodeContents()` 配合 `window.getSelection()` 的选中文本操作在某些情况下与 `contenteditable` 冲突
- **解决**：简化 `attachTextEditHandlers` 中的 dblclick 处理器，只设置 `contenteditable="true"` 和调用 `focus()`，移除 `mousedown` 拦截和文本选择逻辑

#### 问题2：编辑后点击外部/blur时消息未保存
- **现象**：控制台显示 `[edit-bridge] blur, sending text-change:` 日志正常输出，但文件没有保存成功
- **原因**：FileViewer 中的 `onMessage` 函数有 `ev.source !== iframeRef.current?.contentWindow` 检查，由于 iframe 使用 `sandbox="allow-scripts"` 运行，与父页面非同源，导致 `ev.source` 比较失败，消息被直接丢弃
- **解决**：修改 FileViewer.tsx 中的消息来源检查逻辑，对 `od-edit-*` 类型消息（`od-edit-targets`、`od-edit-select`、`od-edit-reorder`、`od-edit-text-change`）放宽来源限制，允许来自 sandboxed iframe 的消息通过

#### 问题3：Docker构建失败
- **现象**：Docker构建在 `corepack enable` 步骤失败
- **原因**：网络问题导致 corepack 准备 pnpm 时超时
- **解决**：重试构建通常可以成功
